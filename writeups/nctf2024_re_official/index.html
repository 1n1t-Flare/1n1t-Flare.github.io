汇编视角下的花指令分析
#

（）花指令更准确来说应该是“垃圾指令”，其目的是干扰反汇编器，尤其是使用线性反汇编算法的工具正常反汇编。其混淆的能力是有限的，而且针对不同反汇编引擎的干扰效果也不尽相同

正因此，花指令的干扰范围仅限静态分析，一旦调试起来它就无能为力了。而静态分析下也有相应的对抗方法，主要是匹配特定的字节码模式（特征）并且nop去除，以帮助反汇编工具正确识别指令乃至进一步反编译。

本题涉及两类常规花指令，

第一类：永恒跳转。ax 异或自身后为0，标志寄存器，，恒定跳转jz分支

xor ax, ax
test ax, ax
jnz offset lable
jz offset lable + 1

第二类基于堆栈的 call +ret。al 经过一系列计算得到一个固定的值，加到 dl 然后 push 到栈上，间接修改了堆栈末尾的返回地址，retf 回去就会改变正常的控制流，跳过部分指令。

call far ptr junkskip

junk segment
junkskip:
    pop dx
    push ax
    xor ax, ax
    ; ...
    add dl, al
    pop ax
    push dx
    retf
junk ends

动调绕过与简化解密
#

动调感觉可以适当结合IDA

DOSBOX自带的调试器DEBUG.EXE，通过单步执行可以准确识别并且反汇编出每一步的指令，但是程序中存在大量循环，而且其不支持断点？？（g cs:xx），只靠step in/over还是太浪费时间，算是美中不足

DOSBox-X的调试器debugbox功能比较完善，虽然说还是会受到花指令干扰，但是结合IDA的交互式反编译还是能弥补这一点。

加密算法是部分魔改的RC4，改动的地方如下：

  • S盒逆序初始化
  • key 左移3位，右移5位
  • 密钥流生成的值 加1

可以写出对应的解密脚本

data = [0x7C, 0x3E, 0x0D, 0x3C, 0x88, 0x54, 0x83, 0x0E, 0x3B, 0xB8, 
        0x99, 0x1B, 0x9B, 0xE5, 0x23, 0x43, 0xC5, 0x80, 0x45, 0x5B, 
        0x9A, 0x29, 0x24, 0x38, 0xA9, 0x5C, 0xCB, 0x7A, 0xE5, 0x93, 
        0x73, 0x0E, 0x70, 0x6D, 0x7C, 0x31, 0x2B, 0x8C]
key = b"NCTf2024nctF"

modikey = [((char<<3)|(char>>5))&0xFF for char in key]
S = [255 - m for m in range(256)]
T = [modikey[n % len(modikey)] for n in range(256)]

j = 0
for i in range(256):
    j = (j + S[i] + T[i]) % 256
    S[i],S[j] = S[j],S[i]
    
i = j = t = 0
for k in range(len(data)):
    i = (i + 1) % 256
    j = (j + S[i]) % 256
    t = (S[i] + S[j]) % 256
    S[i],S[j] = S[j],S[i]
    data[k] ^= (S[t] + 1)

print(bytes(data).decode())

SafeProgram
#

主要考点在CRC和VEH反调试上面，但是加密部分设计的不是很难，所以可能静态分析一下加密函数，不太需要调试就能猜出来。？？

从出题人的视角分析，。。

首先要知道tls回调函数在线程创建或者终止时都会调用。查看导出函数表可以发现 TlsCallback，从这里入手，逐个分析。

第一个tls_callback进行了一些初始化的工作，先注册两个VEH异常处理函数，之后在注册表写入CRC的checksum值。

第二个对代码段进行扫描并且查表计算CRC，和注册表保存的checksum比对，不一致直接RaiseException交给VEH退出程序

另外一个VEH的作用是打乱Sbox，稍后分析

回到main函数。一上来开了新的线程，而且每隔1000ms递归创建新线程，这里目的也就是调用TLS_callback1反复检测CRC。主线程调用Sleep为检测留够时间

后面就是常规的输入-加密-检查过程，要注意的是加密之前，程序主动触发了除零异常，调用第二个VEH handeler

打乱的思路比较简单，Sbox特定索引处的值依次和sbox[0]交换

绕过检测的方法比较多，直接的方法是patch 删去 TLS_CALLBACK1 中调用的CRC检测函数。也可以在调试时只使用硬件断点。

x64dbg，在tls默认断点，修改RIP跳过主线程的crc检测和创建线程的CreateThread ？？

Sleep函数不要单步执行，建议打个断点跳过

后面VEH（还是用x64dbg，IDA debugger行）Shift+F8/F9无视异常，（或者隐藏调试器？可能问题）（调试器接受异常先于VEH，触发异常时直接给VEH）

function Start_Hook(){
    Start_NativeHook("libnative");
    Java.perform(function(){
        var Sec = Java.use("com.nctf.simplelogin.Secure");
        Sec.checkRoot.implementation = function (){
            return false;
        };
        Sec.checkDebug.implementation = function (){
            return false;
        };
    
        var DecStr = Java.use("com.nctf.simplelogin.DecStr");
        //overload('java.lang.String')
        DecStr.get.implementation = function (str) {
            var result = this.get(str);
            console.log(`[*] DecStr.get: ${str}  ${result}`);
            return result;
        };
        //overload('java.lang.String', '[B')
        Sec.doCheck.implementation = function (str,barr) {
            var result = this.doCheck(str,barr);
            console.log(`[*] doCheck: key = ${barr}`);
            return result;
        };
    });
}

function Start_NativeHook(libname) {
    var dlopen = Module.findExportByName(null, "android_dlopen_ext");
    Interceptor.attach(dlopen, {
        onEnter: function (args) {
            var filePath = args[0].readCString();
            if (filePath.indexOf(libname) != -1) {
                console.log(`[+] android_dlopen_ext: start hooking ${libname}`)
                this.isCanHook = true;
            }
        }, onLeave: function (retValue) {
            if (this.isCanHook) {
                this.isCanHook = false;
                hook_native();
            }
        }
    })
}

function hook_native(){
    var target_addr = Module.findBaseAddress("libnative.so").add(0x1F1C);
    Interceptor.attach(target_addr,{
        onEnter: function (args) {
            var key0 = this.context.x22;
            var key1 = this.context.x23;
            console.log(`[+] native key = ${key0} ${key1}`);
        },
        onLeave: function (retval) {}
    });
}

setImmediate(Start_Hook);

gogo
#

（先讲点go逆向知识）

（IDA可能提示packed之类）

主要逻辑是用协程实现了两个并发的寄存器虚拟机，分别加密flag的前后两部分

解题思路依然是还原vm字节码，只要能还原到汇编级别就足以正常分析。

Step.0 vm结构分析
#

在IDA可以找到vm的结构体。前两个好理解，对应寄存器和cache缓存，后面两个是缓冲channel，分别向vm传入字节码和等待返回运行结果，最后一个map是指令集。

【】

runtime_newobject(&RTYPE_main_coroutVM)创建了两个vm struct变量，之后初始化instr，checkres，instrSet字段。

runtime_newproc 开启goroutine，执行 main_main_gowrap，也就是 coroutVM.run()

运行虚拟机

main_main，发现程序将flag拆分成20字节的明文块，分别复制到虚拟机的缓存中。接着同时开启两个vm的协程，并向 instr 管道发送相同的字节码指令

Step.1 指令集逆向
#

从 instr 管道的4字节长度和 handler的参数可以推测出vm使用4字节的定长指令集，看指令名称也可以发现类似ARM。（基本是仿照ARM来设计的，

两个虚拟机的指令集不同，指令集的初始化在 main_init 里面，依次定义了两个map类型变量，指令-函数映射表

（map逆向，）

先runtime_makemap

runtime_mapassign，逐个定义opcode和handler的键值对映射。handler直接根据符号名（推测语义）

指令函数 handler 是二者共用的，需要逆向分析 opcode 和 handler 的对应关系

Step.2 字节码分析
#

off_588470，指向硬编码的字节码。4字节为单位，runtime_chansend1发送到channel

两个虚拟机的指令混在一起，只有能匹配上vm自身指令集的指令会被执行。还原指令时，根据opcode把二者的指令分开会更方便分析。

dump出程序中的vm字节码，写一个自动化或者半自动化的脚本进行还原。这里给出一个可用的 golang 脚本

另外一种，各个handler函数下条件断点，之后类似trace输出。比较省事，一个小问题就是不能区分两个虚拟机的指令。

var InstructionSetB = map[byte]string{
    0x13: "LDR",
    0x14: "LDRI",
    0x17: "STR",
    0x18: "STRI",
    0x2B: "MOV",
    0x91: "ADD",
    0x92: "SUB",
    0x97: "MUL",
    0xC1: "LSL",
    0xC3: "LSR",
    0xCA: "XOR",
    0xCB: "AND",
    0xFE: "RET",
    0xFF: "HLT",
}

func dis(instrSet map[byte]string, bytecode [4]byte) {

    opcode := bytecode[0]
    operands := bytecode[1:]

    if instr, exists := instrSet[opcode]; exists {
        switch instr {
        case "LDR":
            fallthrough
        case "STR":
            fmt.Printf("%s R%d, R%d", instr, operands[0], operands[1])
        case "LDRI":
            fallthrough
        case "STRI":
            fmt.Printf("%s R%d, #%x", instr, operands[0], operands[2])
        case "MOV":
            imm := int32(operands[1]) + int32(operands[2])<<8
            fmt.Printf("%s R%d, #%x", instr, operands[0], imm)
        case "RET":
            fmt.Printf("%s R%d", instr, operands[0])
        case "HLT":
            fmt.Printf("%s", instr)
        default:
            fmt.Printf("%s R%d, R%d, R%d", instr, operands[0], operands[1], operands[2])
        }
        fmt.Print("\n")
    }
}

func disasm(instrSet map[byte]string) {
    var instrcode [4]byte
    data, _ := os.ReadFile("bytecode_dump.bin")
    for i := 0; i < len(data); i += 4 {
        copy(instrcode[:], data[i:i+4])
        dis(instrSet, instrcode)
    }
}

func main(){
    disasm(InstructionSetA)
    disasm(InstructionSetB)
}

Step.3 算法分析
#

如果能顺利还原字节码，那么这道题的难点就解决了。接下来就是根据可读性更好的汇编来分析加密算法。（以第二个虚拟机执行的字节码为例。【】

第一个vm是标准xxtea，第二个是魔改xxtea，只改了移位

两个密钥不同，在字节码中提取

其实特征已经相当明显，看到9e3779b9就已经确定TEA系列，继续向下看移位部分，是xxtea的特征。唯一魔改的地方在于原来标准算法中的左移换成右移，右移换成左移。第一个虚拟机中算法没有改动，是标准xxtea。

字节码虽然看起来很多，但基本上是若干轮循环的重复。两个vm密钥不同，不过都是在前几轮加密中通过MOV指令写入缓存，所以只需要逆前几轮循环，找齐密钥就可以去解密。

（每个虚拟机加密5个uint32明文块,20bytes，也就是n=5，轮数为16
